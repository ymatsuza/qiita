## はじめに

リポジトリを4つ5つと触るようになって、ある日気づいた。

**同じ設定を、何回コピペしてるんだ。**

git の alias を整える。次のリポジトリでもまた整える。Claude Code に「日本語で答えて」と指示する CLAUDE.md を置く。別のリポジトリでもまた置く。マシンを買い替える。`.zshrc` がどこにも残っていない。新しいプロジェクトを始めるたびに、前のリポジトリから設定を漁ってコピーしてくる——。

リポジトリが1個なら、これでいい。コピペは一瞬で終わる。

でも **N個になった瞬間、設定変更のコストが ×N になる**。alias を1個直したいだけなのに、全リポジトリを巡回する羽目になる。これが「複数のGitリポジトリを使っている人ほど dotfiles が効く」理由です。

自分は今、`qiita`・`rcmd`・`GitMD`（Androidアプリ）など複数のリポジトリを、しかも **Mac / Windows / WSL2** をまたいで触っています。その全部を支えているのが、1つの dotfiles リポジトリ。この記事は、**まだ育てている途中のその中身を、丸ごと公開する**ものです。

自分の dotfiles の README には、方針をこう書いてあります。

> - リポジトリ固有でない**個人横断**の設定・メモはここに集約
> - プロジェクトリポジトリには**個人運用ファイルを置かない**

この2行が、複数リポジトリ時代の dotfiles の全てです。

---

## TL;DR

- 複数リポジトリを触るなら、**「個人横断の設定」を1箇所（dotfiles リポジトリ）に集約**する。プロジェクト側には個人運用ファイルを置かない。
- 入れるのは `.zshrc` だけじゃない。**シェル設定・AI（Claude Code）の設定・自動化スクリプト・OS横断設定・「育てる」ログ**まで、個人横断なら全部。
- 効かせ方は**シンボリックリンク**。`~/.claude/CLAUDE.md` などに張れば、全リポジトリ・全マシンに一発で反映。
- マシン固有・秘密は **`~/.zshrc.local`（リポジトリ外）** に逃がす。dotfiles 本体は堂々と公開できる。

## dotfiles リポジトリとは

ホームディレクトリに散らばる「ドットから始まる設定ファイル」（`.zshrc`、`CLAUDE.md` …）を、**1つの Git リポジトリにまとめて管理する**手法です。

やることは単純で、

1. 個人横断の設定・メモを `~/dotfiles/` に集める
2. ホームの本来の場所には**シンボリックリンク**を張る
3. リポジトリを GitHub に push する

これだけ。新しいマシンで `git clone` してリンクを張り直せば、**自分の環境が丸ごと復元**されます。

---

## なぜ「複数リポジトリ」だと効くのか

dotfiles は1リポジトリでも便利ですが、**真価が出るのは複数リポジトリ・複数マシンを横断するとき**です。理由は2つ。

### 1. 設定変更が ×1 で済む

git の alias を1つ追加したいとする。

- **dotfiles なし**：各リポジトリ／各マシンを巡回 → N回編集
- **dotfiles あり**：`~/.zshrc`（=リポジトリ内の実体）を1回編集 → 全リポジトリ・全マシンに反映

リポジトリとマシンが増えるほど、この差は開いていきます。**設定は1箇所、効果は全部**。これが本質です。

### 2. 「どのリポジトリ・どのマシンでも同じ手触り」になる

リポジトリAでは alias が効くのに、Bでは効かない。Mac では `open .` でフォルダが開くのに、WSL では開かない。——この**環境の揺らぎ**が地味に効率を削ります。dotfiles で土台を固定すると、**どこに `cd` しても、どのOSでも同じ手触り**になる。複数リポジトリを高速で行き来する人ほど、この一貫性が効いてきます。

## 大原則：「個人横断」と「プロジェクト固有」を分ける

dotfiles を始めるとき、一番大事なのは技術ではなく**線引き**です。

| 置き場所 | 入れるもの | 例 |
|---|---|---|
| **dotfiles リポジトリ** | 全リポジトリ・全プロジェクトに共通する**個人横断**の設定 | `.zshrc` / global な `CLAUDE.md`・`AGENTS.md` / 自動化スクリプト / 個人メモ |
| **各プロジェクトのリポジトリ** | そのプロジェクト**固有**の設定 | `.editorconfig` / プロジェクト用 `.gitignore` / CI 設定 / プロジェクト用 `CLAUDE.md` |

判断基準はシンプルで、**「新しいリポジトリを作っても変わらないか？」**。変わらないなら個人横断（dotfiles 行き）、プロジェクトごとに違うなら固有（各リポジトリ行き）。

> ⚠️ よくある失敗：プロジェクトリポジトリに個人運用ファイル（自分用のメモ、自分だけが使うスクリプト）を置いてしまう。dotfiles は「**自分という開発者の設定**」であって「プロジェクトの設定」ではありません。ここを混ぜると、複数リポジトリ運用で必ず破綻します。

## 自分の構成（実物）

実際の `~/dotfiles/` は、こうなっています（要点を抜粋）。

```
~/dotfiles/
├── zsh/                 # シェル設定（WSL2 / Linux 向け、Mac 互換を意識）
│   ├── .zshrc
│   └── .zshenv
├── powershell/          # Windows 用プロファイル
│   └── Microsoft.PowerShell_profile.ps1
├── windows/             # Windows セットアップ一式
│   ├── setup-windows.ps1
│   ├── windows-terminal-settings.json
│   └── winget-packages.yaml
├── .claude/             # Claude Code の個人設定
│   ├── CLAUDE.md        #   → ~/.claude/CLAUDE.md にシンボリックリンク
│   └── settings.local.json
├── CLAUDE.md            # 全プロジェクト共通のAI指示（横断）
├── AGENTS.md            # エージェント運用ルール
├── commands/            # Claude Code のカスタムスラッシュコマンド
│   ├── judgment-kata.md
│   ├── monthly-review.md
│   └── reading-mentor.md
├── scripts/             # 自動化スクリプト
│   ├── check_claude_quota.sh
│   ├── claude-scheduled-tasks.sh
│   ├── bootstrap-wsl2.sh
│   └── ...
├── launchagents/        # macOS の常駐ジョブ（launchd）
│   ├── com.ymatsuzaki.claude-quota.plist
│   └── com.ymatsuzaki.judgment-kata.plist
├── judgment-kata-log/   # 「判断の型」練習ログ（育てる）
├── reading-log/         # 読書ログ
├── docs/                # 横断リファレンス（PC設定・手順メモ）
├── logi-backup/         # Logitech Options+ の設定バックアップ
├── README.md
└── .gitignore
```

ポイントは2つ。

**① ツールごとにディレクトリを切っている。** `zsh/`・`.claude/`・`scripts/` …と分けておくと、後から「この設定どこだっけ」が一瞬で済みます。

**② 「設定ファイル」だけじゃない。** dotfiles と聞くと `.zshrc` 置き場を想像しがちですが、自分の場合は **AIの設定・自動化・個人の練習ログ・PC周辺機器の設定まで**、個人横断なものは全部ここに集約しています。「自分という開発者をまるごとバージョン管理する」感覚です。

> 正直に言うと、このリポジトリはまだ**育てている途中**です。README には「`zsh/` `git/` 等の dotfiles 本体は今後追加予定」と書いてあって、もともとは PC設定メモ（`docs/`）や自動化スクリプトから始まり、後からシェル設定を移してきました。**完璧を目指さず、少しずつ移す**——これが続くコツです（後述）。

## カテゴリ別：何を入れているか

### ① シェル設定（zsh / PowerShell）

土台。`.zshrc` には、よく使う alias を全部入れています。複数リポジトリを触るので、**git 系の短縮 alias は特に効きます**。

```bash
# git 短縮 — どのリポジトリでも同じ手で動く
alias g='git'
alias gs='git status'
alias gd='git diff'
alias gl='git log --oneline --graph --decorate'
alias gco='git checkout'

# Docker / npm / pnpm も同様に短縮
alias dc='docker compose'
alias pd='pnpm dev'
```

そして自分の構成で地味に効いているのが、**WSL を Mac に寄せる alias**。

```bash
# WSL 固有 — Mac と同じコマンド名で動くようにする
alias open='explorer.exe'   # Mac の open . 相当
alias pbcopy='clip.exe'     # Mac の pbcopy 相当
bindkey -e                  # Emacs キーバインド（Mac と同じ手触り）
```

Mac と WSL2 を行き来しても `open .` でフォルダが開き、`pbcopy` でクリップボードにコピーできる。**OSが違っても指の動きが変わらない**——これが「同じ手触り」の正体です。Windows ネイティブ側は `powershell/` にプロファイルを、`windows/` に `winget-packages.yaml` まで置いて、セットアップごと再現できるようにしています。

### ② AI（Claude Code）の設定

ここが今っぽいところ。**全プロジェクト共通のAI指示**を dotfiles で一元管理しています。

- `CLAUDE.md` / `.claude/CLAUDE.md` → `~/.claude/CLAUDE.md` にシンボリックリンク。「日本語で答えて」「コミットメッセージはこの形式で」のような**横断的な好み**を1箇所に。
- `AGENTS.md` → エージェント運用のルール。
- `commands/` → Claude Code の**カスタムスラッシュコマンド**（`/judgment-kata`、`/monthly-review`、`/reading-mentor` など）。

新しいリポジトリで Claude Code を起動した瞬間、**もう自分の流儀を知っている**状態になる。プロジェクト固有の指示は各リポジトリの `CLAUDE.md` に、横断的な指示は dotfiles に——という線引きが、複数リポジトリ × AI 開発では効きます。

### ③ 自動化（scripts / launchagents）

`scripts/` に運用スクリプト、`launchagents/` に macOS の常駐ジョブ（launchd）を置いています。たとえば Claude の使用量を監視する `check_claude_quota.sh` を `launchagents/com.ymatsuzaki.claude-quota.plist` で定期実行する、といった具合。**「マシンの自動化そのもの」も個人横断の資産**なので、dotfiles に入れて新マシンでも復元できるようにしています。

### ④ 「育てる」ログ

`judgment-kata-log/`（設計判断・コードレビューの練習ログ）と `reading-log/`（読書ログ）。設定ファイルではありませんが、**個人横断で、かつ日々追記して育てるもの**なので dotfiles の住人です。

このあたりは、以前書いた「[シェル履歴を『育てる』CLIツール rcmd](https://qiita.com/ymatsuza/items/44049f6fd43c6e6bef58)」と同じ発想で、**手元の知識を Git で育てていく**ための置き場になっています。

## 管理方法：シンボリックリンク

dotfiles の管理ツールは GNU Stow や chezmoi などもありますが、自分は**素のシンボリックリンク**でやっています。理由は「依存を増やしたくない」「仕組みが透明で壊れても直せる」から。

たとえば Claude の設定なら、こう張るだけ。

```bash
ln -snfv ~/dotfiles/.claude/CLAUDE.md ~/.claude/CLAUDE.md
ln -snfv ~/dotfiles/zsh/.zshrc         ~/.zshrc
```

シンボリックリンクなので、**リンク先（`~/.zshrc`）を編集すれば実体（リポジトリ内のファイル）が変わる**。あとは `git add -p && git commit && git push` で同期されます。「設定をいじる＝dotfiles を育てる」が自然に回るのが、リンク方式の良いところ。

新しいマシンでの復元は、`git clone` してリンクを張り直すだけ。リンク対象が増えてきたら、**張る処理を1本のスクリプトにまとめておく**と一発で済みます（自分も `zsh/` を移したタイミングで整備中）。

## 秘密情報・マシン固有の扱い

dotfiles を public で公開するなら、**秘密情報とマシン固有の値を本体に入れない**のが鉄則です。自分のやり方はシンプルで、`.zshrc` の末尾でこれだけ。

```bash
# ── ローカル上書き ──
[[ -f "$HOME/.zshrc.local" ]] && source "$HOME/.zshrc.local"
```

API キーや、そのマシンだけのパス設定は **`~/.zshrc.local`（リポジトリの外）** に書く。公開する dotfiles 本体には一切入りません。「**公開できる設定**」と「**そのマシン/秘密だけの設定**」を物理的に分けておけば、dotfiles を堂々と public にできます。

リポジトリの `.gitignore` も最小限で、同期したくない作業ディレクトリだけ除外しています。

```bash
# .gitignore
.claude/worktrees/
.superpowers/
```

## まだ入れていないが、複数リポジトリなら効くもの

正直に「これから入れたい」リストも。複数リポジトリ運用なら、定番として効くやつです。

- **global gitignore**：`.DS_Store` や `.env` など「全リポジトリで無視したいゴミ」を `~/.gitignore_global` に1回書いて、`git config --global core.excludesfile` で有効化。各リポジトリの `.gitignore` に毎回書く ×N を消せます。
- **`ghq` + `fzf`**：clone を `ghq` 経由に統一すると置き場が機械的に決まり、**リポジトリ迷子**が消える。`ghq list --full-path | fzf` で全リポジトリにジャンプ。リポジトリ数が二桁なら一番効く小技です。
- **`core.hooksPath`**：全リポジトリ共通の git hook（コミット前 lint、秘密情報の検出）を1箇所で共有。

自分の dotfiles はまだここに手が回っていませんが、**多リポジトリの人にはこの3つを特におすすめ**します。

## ハマったところ・運用のコツ

- **いきなり完璧を目指さない。** 自分の dotfiles も、最初は PC設定メモと自動化スクリプトだけ。`.zshrc` を移したのは後から。「全部移すぞ」と意気込むと挫折します。**今いちばん ×N で困っているもの1個**から移すのが正解。
- **リンク方式は「既存ファイルの退避」を忘れがち。** `ln` を走らせる前に既存の `~/.zshrc` を上書きして泣くやつ。先にバックアップを取るか、`ln -snf` の前にリネームしておくと安全。
- **OS差は alias とガードで吸収する。** Mac / WSL / Windows を1リポジトリで回すなら、`open`/`pbcopy` のような alias で名前を揃え、OS依存は分岐 or `.zshrc.local` に逃がす。
- **マシン固有は `.zshrc.local` へ。** 本体に `if [[ $(hostname) == ... ]]` を増やし始めると地獄。マシン差は本体の外（`.zshrc.local`）で吸収するのが綺麗。

## まとめ

dotfiles リポジトリの効果は、**リポジトリの数 × マシンの数**に比例して大きくなります。

- 1リポジトリ・1マシンなら、コピペで十分。
- **複数リポジトリを Mac / Windows / WSL で行き来するなら、設定変更が ×N か ×1 かの分かれ道。**

そして dotfiles に入れるのは、もう `.zshrc` だけじゃありません。**シェルも、AIへの指示も、自動化も、日々育てるログも**——「個人横断でつねに連れて歩きたいもの」は全部入れていい。自分のリポジトリは、いつのまにか「設定置き場」から「**自分という開発者の司令塔**」に育っていました。

やることは地味です。設定を `~/dotfiles/` に集めて、リンクを張って、push するだけ。難しい技術は何も出てきませんでした。出てきたのは `ln -s` と git と、ちょっとした alias だけ。

複数リポジトリを行き来しているなら、今日 `~/dotfiles` を `git init` するところから。まずは、いま一番コピペしている設定1個から始めてみてください。
