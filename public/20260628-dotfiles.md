## はじめに

リポジトリを4つ5つと触るようになって、ある日気づいた。

**同じ設定を、何回コピペしてるんだ。**

`.gitignore` に `.DS_Store` を足す。次のリポジトリでもまた足す。git の alias を増やす。マシンを買い替える。`.zshrc` がどこにも残っていない。新しいプロジェクトを始めるたびに、前のリポジトリから設定を漁ってコピーしてくる——。

リポジトリが1個なら、これでいい。コピペは一瞬で終わる。

でも **N個になった瞬間、設定変更のコストが ×N になる**。alias を1個直したいだけなのに、全リポジトリを巡回する羽目になる。これが「複数のGitリポジトリを使っている人ほど dotfiles が効く」理由です。

自分は今、`qiita`・`rcmd`・`GitMD`（Androidアプリ）・その他いくつかのリポジトリを `git worktree` で行き来しながら開発しています。この記事は、そのために組んだ **dotfiles リポジトリの実際の構成**を、丸ごと公開するものです。

---

## TL;DR

- 複数リポジトリを触るなら、**全リポジトリ共通の設定**を1箇所（dotfiles リポジトリ）に集約する。
- 鍵は「**どこに何を置くか**」の線引き。**ホーム配下の横断設定は dotfiles**、**プロジェクト固有の設定は各リポジトリ**。
- 管理はシンボリックリンク + `install.sh` で十分。新マシンは `git clone && ./install.sh` で復元。
- 複数リポジトリ運用に特に効くのは **`ghq` + `fzf`**、**global gitignore**、**`core.hooksPath`**、**global な `CLAUDE.md`** の4つ。

## dotfiles リポジトリとは

ホームディレクトリに散らばる「ドットから始まる設定ファイル」（`.zshrc`、`.gitconfig`、`.vimrc` …）を、**1つの Git リポジトリにまとめて管理する**手法です。

やることは単純で、

1. 設定ファイルを `~/dotfiles/` の下に集める
2. ホームの本来の場所には**シンボリックリンク**を張る
3. リポジトリを GitHub に push する

これだけ。あとは新しいマシンで `git clone` してリンクを張り直せば、**自分の環境が丸ごと復元**されます。

---

## なぜ「複数リポジトリ」だと効くのか

dotfiles は1リポジトリでも便利ですが、**真価が出るのは複数リポジトリを横断するとき**です。理由は2つ。

### 1. 設定変更が ×1 で済む

git の alias を1つ追加したいとする。

- **dotfiles なし**：各リポジトリの `.git/config` を巡回 → N回編集
- **dotfiles あり**：`~/.gitconfig` を1回編集 → 全リポジトリに即反映

リポジトリが増えるほど、この差は開いていきます。**設定は1箇所、効果は全リポジトリ**。これが本質です。

### 2. 「どのリポジトリでも同じ手触り」になる

リポジトリAでは alias が効くのに、Bでは効かない。マシンを変えたら fzf の挙動が違う。——この**環境の揺らぎ**が地味に効率を削ります。dotfiles で土台を固定すると、**どこに `cd` しても同じ手触り**になる。複数リポジトリを高速で行き来する人ほど、この一貫性が効いてきます。

## 大原則：「共通」と「固有」を分ける

dotfiles を始めるとき、一番大事なのは技術ではなく**線引き**です。

| 置き場所 | 入れるもの | 例 |
|---|---|---|
| **dotfiles リポジトリ**（ホーム配下） | 全リポジトリ・全プロジェクトに共通する横断設定 | `.zshrc` / `~/.gitconfig` / global gitignore / 共通の git hooks / global な `CLAUDE.md` |
| **各プロジェクトのリポジトリ** | そのプロジェクト固有の設定 | `.editorconfig` / プロジェクト用 `.gitignore` / CI 設定 / プロジェクト用 `CLAUDE.md` |

判断基準はシンプルで、**「新しいリポジトリを作っても変わらないか？」**。変わらないなら共通（dotfiles 行き）、プロジェクトごとに違うなら固有（各リポジトリ行き）。

> ⚠️ よくある失敗：プロジェクト固有のものまで dotfiles に入れてしまう。dotfiles は「**自分という開発者の設定**」であって「プロジェクトの設定」ではありません。ここを混ぜると、すぐに破綻します。

## 自分のディレクトリ構成

実際の `~/dotfiles/` はこうなっています。

```
~/dotfiles/
├── install.sh           # シンボリックリンクを一括で張る
├── Makefile             # make install / make update のショートカット
├── git/
│   ├── .gitconfig       # user / alias / worktree ヘルパ / ghq root
│   ├── .gitignore_global# OS・エディタ・秘密ファイルを全リポジトリで無視
│   └── hooks/           # core.hooksPath で全リポジトリ共通のフック
├── zsh/
│   ├── .zshrc
│   ├── .zshenv
│   └── aliases.zsh
├── claude/
│   └── CLAUDE.md        # ~/.claude/CLAUDE.md（全プロジェクト共通のAI指示）
├── rcmd/
│   └── store.toml       # 自作 rcmd の共有コマンド集（$RCMD_STORE）
├── config/
│   ├── starship.toml
│   └── direnv/
└── .gitignore           # secrets/ など同期したくないものを除外
```

ポイントは、**ツールごとにディレクトリを切っている**こと。`git/`・`zsh/`・`claude/` …と分けておくと、後から「git 周りだけ見たい」「この設定どこだっけ」が一瞬で済みます。

## 管理方法：シンボリックリンク + install.sh

dotfiles の管理ツールは GNU Stow や chezmoi などいろいろありますが、自分は**素の `install.sh`** に落ち着きました。理由は「依存を増やしたくない」「新マシンで `git clone` 直後に何もインストールせず動かしたい」から。

中身は驚くほど単純です。

```bash
#!/usr/bin/env bash
set -euo pipefail

DOTFILES="$HOME/dotfiles"

# ホーム直下に張るリンク（リポジトリ内パス:リンク先）
link() {
  local src="$DOTFILES/$1"
  local dst="$2"
  mkdir -p "$(dirname "$dst")"
  ln -snfv "$src" "$dst"   # -n: リンク先がディレクトリでも上書き, -f: 既存を置換
}

link zsh/.zshrc            "$HOME/.zshrc"
link zsh/.zshenv           "$HOME/.zshenv"
link git/.gitconfig        "$HOME/.gitconfig"
link git/.gitignore_global "$HOME/.gitignore_global"
link claude/CLAUDE.md      "$HOME/.claude/CLAUDE.md"
link config/starship.toml  "$HOME/.config/starship.toml"

echo "✅ dotfiles linked."
```

新しいマシンでの復元は、これだけ。

```bash
git clone https://github.com/ymatsuza/dotfiles.git ~/dotfiles
cd ~/dotfiles && ./install.sh
```

シンボリックリンクなので、**リンク先を編集すれば実体（リポジトリ内のファイル）が変わる**。あとは `git add -p && git commit && git push` で同期されます。「設定をいじる＝dotfiles を育てる」が自然に回るのが、リンク方式の良いところ。

## 複数リポジトリ運用に「特に」効く4つ

ここからが本題。複数の Git リポジトリを使う人にとって、dotfiles に入れておくと効果が大きいものを4つ挙げます。

### ① `ghq` + `fzf` ──「リポジトリが多すぎて行方不明」を解決する

複数リポジトリ運用の最大の悩みは、**clone したリポジトリがどこにあるか分からなくなる**こと。`~/work`、`~/dev`、`~/ghq`、デスクトップ…と散らばっていく。

[`ghq`](https://github.com/x-motemen/ghq) は、これを**ホスト/オーナー/リポジトリ名で一意なパスに強制**してくれるツールです。

```bash
# clone は ghq 経由で
ghq get ymatsuza/rcmd
# → ~/ghq/github.com/ymatsuza/rcmd に必ず置かれる
```

置き場所が機械的に決まるので、**もう迷子になりません**。そして、`fzf` と組み合わせて「リポジトリ間ジャンプ」を作るのが定番。`.zshrc`（つまり dotfiles）にこの関数を入れています。

```bash
# Ctrl+] で全リポジトリを fzf 選択して即 cd
g() {
  local dir
  dir=$(ghq list --full-path | fzf) && cd "$dir"
}
zle -N g
bindkey '^]' g   # Ctrl+] でインタラクティブに飛ぶ
```

リポジトリが20個あっても、`Ctrl+]` → 数文字打つ → Enter で目的のリポジトリに着地。**複数リポジトリ運用は、この一発で世界が変わります。**

`.gitconfig` には ghq の root だけ書いておけば OK。

```ini
[ghq]
  root = ~/ghq
```

### ② global gitignore ──「全リポジトリで同じゴミを無視」を1回で

`.DS_Store`、`*.swp`、`.env`、`.idea/`、`__pycache__/`。これらは**どのリポジトリでも無視したい**けど、プロジェクト固有ではない。各リポジトリの `.gitignore` に毎回書くのは典型的な ×N の無駄です。

global gitignore に1回書けば、全リポジトリに効きます。

```bash
# git/.gitignore_global
.DS_Store
*.swp
*~
.env
.env.local
.idea/
.vscode/
__pycache__/
.venv/
```

これを `.gitconfig` で有効化（dotfiles に入れておく）。

```ini
[core]
  excludesfile = ~/.gitignore_global
```

> 線引きの注意：**プロジェクト固有**の無視ルール（ビルド成果物のパスなど）は、各リポジトリの `.gitignore` に書く。global gitignore は「**自分の環境/エディタが吐くゴミ**」だけに留めるのがコツ。チームに `.idea/` を押し付けないためにも、エディタ系は global 側が筋です。

### ③ `core.hooksPath` ──「全リポジトリ共通の git hook」

「コミット前に必ず lint」「秘密情報のコミットを弾く」——こういうフックを**全リポジトリで共有**したいとき。従来は各リポジトリの `.git/hooks/` にコピーするしかなく、これも ×N でした。

`core.hooksPath` を使うと、**1箇所のフックを全リポジトリで使い回せます**。

```ini
[core]
  hooksPath = ~/dotfiles/git/hooks
```

```bash
# git/hooks/pre-commit （実行権限を付けておく）
#!/usr/bin/env bash
# 秘密っぽい文字列をうっかりコミットしないための最後の砦
if git diff --cached | grep -E "(AKIA[0-9A-Z]{16}|-----BEGIN.*PRIVATE KEY-----)"; then
  echo "🚫 秘密情報らしき文字列が含まれています。コミットを中止しました。"
  exit 1
fi
```

複数リポジトリで「秘密情報の事故」を一律で防げるのは、想像以上に安心感があります。

### ④ global な `CLAUDE.md` ──「AIへの指示」も dotfiles に

AI コーディング（Claude Code など）を全リポジトリで使うなら、**全プロジェクト共通のAI指示**は `~/.claude/CLAUDE.md` に置けます。プロジェクト固有の指示は各リポジトリの `CLAUDE.md` に、横断的な好み（「日本語で答えて」「コミットメッセージはこの形式で」など）は dotfiles 側に。

```markdown
<!-- claude/CLAUDE.md → ~/.claude/CLAUDE.md にリンク -->
- 回答は簡潔に。結論から書く。
- コミットメッセージは Conventional Commits 形式で。
- 不確実なときは、自信度を明示する。
```

これも「設定は1箇所、効果は全リポジトリ」の好例。新しいリポジトリで Claude Code を起動した瞬間、**もう自分の流儀を知っている**状態になります。

> ちなみに自作の CLI ツール [rcmd](https://qiita.com/ymatsuza/items/44049f6fd43c6e6bef58) のコマンド集（`store.toml`）も、`$RCMD_STORE=~/dotfiles/rcmd/store.toml` を `.zshrc` に書いて dotfiles で同期しています。「複数マシン × 複数リポジトリ」でも、よく使うコマンドが全部ついてくる。

## 秘密情報をどう扱うか

dotfiles を public で公開するなら、**秘密情報を絶対に入れない**のが鉄則です。やり方は2つ。

1. **`.gitignore` で除外する**：`secrets/` ディレクトリを切って、`.gitignore` で丸ごと無視。リンクは張るが中身は同期しない。
   ```bash
   # dotfiles の .gitignore
   secrets/
   *.local
   ```
2. **`.zshrc` の末尾で外部ファイルを読む**：API キーなどは `~/.secrets.zsh`（git 管理外）に書いて、こう読み込む。
   ```bash
   # .zshrc の末尾
   [ -f ~/.secrets.zsh ] && source ~/.secrets.zsh
   ```

「公開できる設定」と「公開できない秘密」を物理的に分けておけば、dotfiles を堂々と public にできます。

## ハマったところ・運用のコツ

実際に運用してわかった落とし穴も正直に。

- **いきなり完璧を目指さない。** 最初は `.zshrc` と `.gitconfig` の2つだけで十分。育てながら足す。「全部移すぞ」と意気込むと、たいてい挫折します。
- **リンク方式は「既存ファイルの退避」を忘れがち。** `install.sh` を走らせる前に、既存の `~/.zshrc` を上書きして泣くやつ。`ln` の前にバックアップを取るか、`-b` オプションを噛ませておくと安全。
- **OS差はガードで吸収する。** macOS と Linux で `.zshrc` を共有するなら、`if [[ "$OSTYPE" == darwin* ]]; then ...` で分岐。1ファイルで両対応できます。
- **`git/.gitconfig` の `[user]` は環境で変えたいことがある。** 仕事用と個人用でメールアドレスを分けるなら、`includeIf` でディレクトリ別に切り替えると綺麗。
  ```ini
  [includeIf "gitdir:~/ghq/github.com/会社org/"]
    path = ~/dotfiles/git/.gitconfig-work
  ```
  これも「複数リポジトリ」ならではの便利技です。

## まとめ

dotfiles リポジトリの効果は、リポジトリの数に比例して大きくなります。

- 1リポジトリなら、コピペで十分。
- **5リポジトリなら、設定変更が ×5 か ×1 かの分かれ道。**
- 10リポジトリを超えたら、dotfiles 無しはもう正気じゃない。

やることは地味です。設定を `~/dotfiles/` に集めて、リンクを張って、push するだけ。難しい技術は何も出てきませんでした。出てきたのは `ln -s` と git と、ちょっとした `.zshrc` の関数だけ。

でもこの地味な一手で、**「どのリポジトリでも、どのマシンでも、同じ自分の環境」**が手に入ります。複数リポジトリを行き来しているなら、今日 `~/dotfiles` を `git init` するところから始めてみてください。

まずは `.zshrc` 1枚から。それで十分です。
